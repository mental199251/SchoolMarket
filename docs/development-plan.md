# “校淘空间”校园二手交易平台开发计划

## 1. 计划依据

本计划依据《概要设计报告》和《详细设计报告》制定，并结合当前仓库状态安排实施顺序。

当前基线：

- `frontend/` 已使用官方 uni-app Vue3/Vite JavaScript 模板初始化。
- 前端依赖已经安装，微信小程序开发构建产物已经存在。
- 当前只有官方首页示例，尚未建立业务页面、API 封装和通用组件。
- 尚无 `server/`、自动化测试、接口文档、初始化数据和运行说明。

系统采用 uni-app + Vue3 + Vite、Flask、MongoDB、JWT 和 Ollama。课程阶段不实现真实支付、物流配送和第三方担保，实际交付在线下完成，平台只管理商品、交易意向及状态。

## 2. 开发目标与范围

### 2.1 P0 核心闭环

P0 必须优先完成，并能够连续演示：

1. 前后端工程骨架和健康检查。
2. 用户注册、登录、退出、资料维护和密码修改。
3. 商品发布、图片上传、编辑、下架、浏览、搜索、筛选和详情。
4. 买家发起购买请求。
5. 卖家确认或取消请求。
6. 交易参与者标记线下交易完成。
7. 交易完成后商品同步变为 `sold`。
8. JWT、角色和资源归属校验。

### 2.2 P1 管理与通知

- 管理员用户禁用和恢复。
- 管理员商品治理。
- 交易状态消息提醒和消息已读。
- 管理员操作日志。
- 基础公告管理。

### 2.3 P2 展示增强

- 平台概览、热门分类和活跃用户统计。
- Ollama 商品标题建议和描述生成。
- AI 调用日志、超时和失败降级。
- 页面体验、性能和答辩演示优化。

若工期缩短，只允许削减 P2，然后削减 P1 中的公告功能，不得破坏 P0 主流程。

## 3. 编码前冻结的设计口径

开发开始前应把下列内容写入 `docs/api.md` 和 `docs/enums.md`，前后端共同遵守：

1. 所有业务接口统一使用 `/api/v1` 前缀。详细设计接口表中的 `/auth/*`、`/products/*` 均视为省略了此前缀。
2. 数据库角色只保留 `user` 和 `admin`。买家、卖家是同一普通用户在不同业务中的身份，不建立互斥角色。
3. 第一版关闭商品审核，`PRODUCT_REVIEW_ENABLED=false`，商品提交后直接进入 `available`；保留 `pending_review` 和 `rejected` 供后续启用。
4. 商品状态统一为 `draft`、`pending_review`、`available`、`off_shelf`、`rejected`、`sold`。
5. 交易状态统一为 `pending`、`confirmed`、`cancelled`、`completed`。
6. 统一响应为 `{ success, data, message, error_code }`；分页数据必须包含 `items`、`total`、`page`、`page_size`。
7. 时间使用带时区的 ISO 8601 字符串；金额在 MongoDB 中统一使用 Decimal128 或整数分，禁止直接使用二进制浮点数存储价格。
8. 受保护接口每次读取用户最新状态，禁用用户即使持有旧 Token 也返回 `USER_DISABLED`。
9. 第一版不做微信授权绑定；先完成账号密码认证。微信绑定作为独立扩展，不阻塞主流程。
10. 交易完成接口必须使用条件更新防止重复完成；若 MongoDB 未使用副本集事务，则采用补偿日志处理“交易已完成但商品未标记 sold”的异常。

## 5. 里程碑与实施顺序

### M0：基线验证与协作规范

预计工作量：1 人日。

任务：

- 验证 `npm ci`、H5 构建和微信小程序构建。
- 固定 Node.js、Python 和依赖版本。
- 建立分支、提交和接口变更规范。
- 创建 README、环境变量示例和本地运行说明。
- 确定开发端口、MongoDB 数据库名和上传目录。

退出条件：

- 新环境按 README 可完成依赖安装和微信小程序构建。
- 不依赖个人机器中的绝对路径或硬编码密钥。

### M1：后端骨架与前后端连通

预计工作量：2 人日。

后端任务：

- 建立 Flask 应用工厂、配置加载、PyMongo 和 CORS。
- 建立统一响应、异常基类、请求标识和日志。
- 实现 `GET /health` 和 `GET /ready`。
- `/health` 只判断 Flask 存活，`/ready` 检查 MongoDB；Ollama 不作为核心就绪条件。

前端任务：

- 建立 `api/config.js`、`api/http.js` 和 `api/index.js`。
- 封装 `uni.request` 和 `uni.uploadFile`。
- 统一处理超时、网络失败、401、403、409 和 503。
- 首页显示后端健康检查结果。

测试：

- Flask 健康检查测试。
- MongoDB 正常和不可用场景。
- 微信开发者工具到 Flask 的真实链路测试。

退出条件：

```text
微信小程序页面 -> uni.request -> Flask -> JSON -> 页面状态
```

### M2：认证、权限与用户资料

预计工作量：4 人日。

接口：

- `POST /api/v1/auth/register`
- `POST /api/v1/auth/login`
- `POST /api/v1/auth/logout`
- `PUT /api/v1/auth/password`
- `GET /api/v1/users/me`
- `PUT /api/v1/users/me`

后端任务：

- 建立 `users` 集合、账号唯一索引和用户状态索引。
- 密码使用 bcrypt 或 Argon2 哈希。
- 实现 JWT 签发、解析、过期和禁用用户检查。
- 普通用户提交的 `role`、`status` 等权限字段必须被忽略或拒绝。
- 初始化普通用户 A、普通用户 B 和管理员账号。

前端任务：

- 登录、注册、个人中心、资料编辑和修改密码页面。
- 保存 Token、用户摘要和过期时间。
- 请求自动携带 Authorization；401 时一次性清理登录态。

退出条件：

- 注册、登录、查看资料、修改资料、修改密码和退出均可演示。
- 重复账号、错误密码、过期 Token、禁用账号和越权访问测试通过。

### M3：商品浏览、发布与图片上传

预计工作量：6 人日。

接口：

- `POST /api/v1/uploads/images`
- `POST /api/v1/products`
- `GET /api/v1/products`
- `GET /api/v1/products/{id}`
- `PUT /api/v1/products/{id}`
- `DELETE /api/v1/products/{id}`
- `PUT /api/v1/products/{id}/status`

后端任务：

- 建立 `products`、`categories` 集合及设计报告要求的索引。
- 实现公开列表、关键词、分类、成色、价格、排序和分页。
- 公开查询默认只返回 `available`。
- 实现商品创建、编辑、下架、恢复和删除的资源归属校验。
- 上传服务校验 MIME、扩展名、数量和大小，使用随机文件名并只返回 URL。
- 初始化教材资料、电子产品、生活用品和运动器材分类及示例商品。

前端任务：

- 首页、商品列表、详情、发布、编辑和“我的商品”。
- 商品卡片、筛选栏、状态标签、图片上传和空状态组件。
- 筛选变化后回到第一页，失败时保留条件和表单内容。

退出条件：

- 用户可上传图片并发布商品，另一用户能搜索并查看详情。
- 用户不能编辑他人商品，已售或下架商品不能被公开购买。
- 列表与详情在测试数据规模下 2 秒内返回。

### M4：交易请求与核心状态机

预计工作量：5 人日。

接口：

- `POST /api/v1/trades`
- `GET /api/v1/trades/my-buy`
- `GET /api/v1/trades/my-sell`
- `PUT /api/v1/trades/{id}/confirm`
- `PUT /api/v1/trades/{id}/cancel`
- `PUT /api/v1/trades/{id}/complete`

后端任务：

- 建立 `trade_requests` 集合和买家、卖家、商品、状态索引。
- 禁止购买自己的商品。
- 阻止同一买家对同一商品创建重复有效请求。
- 使用条件更新实现 `pending -> confirmed -> completed` 和 `pending -> cancelled`。
- 完成交易时同步更新商品为 `sold`，记录完成时间和操作者。
- 为冲突返回最新状态，映射为 HTTP 409。

前端任务：

- “我的购买”和“收到的请求”页面。
- 按状态显示确认、取消和完成按钮。
- 危险操作二次确认；提交中禁止重复点击；成功后刷新状态。

退出条件：

- 普通用户 A 发布商品，用户 B 发起购买，A 确认，任一参与者完成，商品变为已售。
- 自购、重复请求、非卖家确认、非参与者完成和重复完成均被后端拒绝。

### M5：消息、管理员治理和操作日志

预计工作量：5 人日。

任务：

- 建立 `messages` 和 `operation_logs` 集合及索引。
- 交易创建、确认、取消和完成时生成站内消息。
- 实现消息列表和消息已读接口。
- 实现管理员用户列表、禁用和恢复。
- 实现管理员商品治理、公告维护和日志查询。
- 所有治理操作记录操作者、目标、动作、时间和必要详情。
- 前端实现消息页及管理员用户、商品、公告和日志页面。

退出条件：

- 普通用户不能进入或调用管理员功能。
- 禁用用户的旧 Token 立即失效。
- 管理员关键操作可通过日志追溯。

### M6：统计报表

预计工作量：3 人日。

接口：

- `GET /api/v1/stats/overview`
- `GET /api/v1/stats/categories`
- `GET /api/v1/stats/users`

任务：

- 使用 MongoDB aggregate 统计发布量、成交量、热门类别和活跃用户。
- 限定统计时间范围和结果字段。
- 前端实现管理端统计卡片和图表。
- 数据量增加后再启用 `statistics_cache`，第一版不提前缓存。

退出条件：

- 统计结果能用固定测试数据人工核对。
- 统计接口在测试数据规模下 5 秒内返回，异常不影响普通用户交易。

### M7：Ollama AI 辅助发布

预计工作量：3 人日。

接口：

- `POST /api/v1/ai/title`
- `POST /api/v1/ai/description`

任务：

- 建立独立 AIService，模型名称和地址从环境变量读取。
- Prompt 强调校园二手、真实、简洁和禁止虚构。
- 第一版采用非流式调用并配置连接、响应超时。
- AI 只返回候选内容，不直接写入商品。
- 建立 `ai_generation_logs`，只保存必要摘要、模型、耗时和错误。
- 发布页面允许采用、编辑或放弃建议。

退出条件：

- Ollama 正常时能生成标题和描述建议。
- 关闭 Ollama 后返回 `AI_UNAVAILABLE`，原表单内容保留，用户仍可手工发布。

### M8：系统测试、性能与答辩交付

预计工作量：5 人日。

任务：

- 完成后端单元测试、接口测试和状态机测试。
- 完成前端请求封装、表单校验和主要页面测试。
- 在微信开发者工具完成全链路回归。
- 检查账号、文本、价格、状态和上传安全。
- 检查接口响应时间、重复提交和状态冲突。
- 准备初始化脚本、数据库备份说明、测试报告和演示脚本。
- 更新 README、接口文档、环境说明和部署说明。

最终验收路径：

```text
用户 A 注册并发布商品
-> 用户 B 搜索并发起购买
-> 用户 A 确认请求
-> 参与者完成线下交易
-> 商品变为已售
-> 双方查看消息和交易记录
-> 管理员查看日志与统计
-> AI 可用和不可用两种发布场景
```

## 7. 协作方式

- 接口负责人先提交接口契约、错误码和 mock 响应，前端再按契约开发。
- 每个模块按“接口契约 -> 后端测试 -> 后端实现 -> 前端页面 -> 联调 -> 验收”推进。
- 前端、后端和数据库共同维护 `docs/enums.md`，禁止在页面中自行创造状态值。
- 一个合并请求只处理一个可验证模块，必须包含测试或明确的手工验证记录。
- 接口、字段、枚举或状态机变化时，代码与文档必须在同一提交中更新。
- 每周至少完成一次主流程演示，不把全部联调集中到最后一周。

## 8. 测试矩阵

### 8.1 自动化测试

- AuthService：注册、登录、密码哈希、禁用、Token 过期。
- ProductService：资源归属、筛选、分页和状态变更。
- UploadService：类型、大小、伪造扩展名和路径安全。
- TradeService：合法流转、自购、重复请求、越权和并发冲突。
- Admin/Stats：角色权限、日志写入和聚合正确性。
- AIService：成功、超时、不可用和非法返回。
- API：响应结构、HTTP 状态和错误码一致性。

### 8.2 手工测试

- 微信开发者工具中的注册、登录、商品、交易和管理流程。
- 真机访问局域网后端时的地址配置和上传。
- MongoDB 停止、Ollama 停止、网络超时和 Token 过期。
- 空列表、慢请求、重复点击和旧页面状态冲突。

## 9. 完成定义

每个模块只有同时满足以下条件才算完成：

1. 接口和状态规则已记录在文档中。
2. 后端实现具备参数、权限和资源归属校验。
3. 自动化测试通过，关键失败路径有覆盖。
4. 前端具备加载、空数据、成功、失败和重试状态。
5. 微信开发者工具真实联调通过。
6. 不在代码中保存密码、JWT 密钥、MongoDB 凭据或 Ollama 地址。
7. 相关 README、接口文档和测试记录已同步更新。

## 10. 主要风险与控制

| 风险 | 控制措施 |
| --- | --- |
| 前后端接口口径不一致 | 先冻结 `/api/v1`、响应结构、分页和错误码，接口变更同步文档。 |
| 买家/卖家角色理解错误 | 数据库仅使用 user/admin，买卖权限按资源归属判断。 |
| 交易状态被重复或越权修改 | 状态机集中在 TradeService，使用条件更新和 HTTP 409。 |
| 完成交易的两次写入不一致 | 优先使用事务；不具备事务条件时记录补偿日志并提供管理员核对流程。 |
| 图片上传带来安全和存储问题 | 双重校验类型和大小，随机文件名，隔离上传目录，只返回 URL。 |
| AI 服务拖慢或阻断发布 | 独立服务、严格超时、503 映射，AI 失败时继续手工发布。 |
| 最后阶段才开始联调 | 每个里程碑都设置真实前后端退出条件，每周走查完整主流程。 |
| 答辩环境数据或依赖缺失 | 提供固定账号、初始化脚本、示例数据、备份说明和启动检查清单。 |
